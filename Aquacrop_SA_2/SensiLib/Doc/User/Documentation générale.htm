Introduction

 

SENSILIB est une bibliothèque de fonctions Matlab pour l’analyse de sensibilité de modèles de simulation. Nous entendons ici par analyse de sensibilité l’étude des conséquences de perturbations sur tout ou partie des facteurs d’entrée du modèle considéré sur tout ou partie de ses variables de sortie. En pratique cette étude se caractérise par le calcul d’indices de sensibilité qui permettent d’analyser l’importance respective des différents facteurs d’entrée considérés pour la simulation de la (des) variable(s) de sortie considérée(s), de façon qualitative (simple hiérarchie) ou quantitative (hiérarchie avec ordre de grandeur des écarts). Il existe différentes classes de méthodes (méthodes de screening, méthodes locales, méthodes globales) dont les objectifs et les indices sont différents et parmi ces classes, en général, plusieurs types de méthodes.

 

Il est important de noter que l’utilisation des fonctions de cette bibliothèque nécessite la connaissance des concepts de base de l’analyse de sensibilité, en particulier des caractéristiques de la famille à laquelle appartient la méthode utilisée, et la définition claire des objectifs poursuivis. Le lecteur pourra se référer à Saltelli et al (2000)[1] pour une introduction complète à l’analyse de sensibilité.

 

Cette bibliothèque a été conçue pour fournir des outils d’analyse de sensibilité utilisables sur le modèle STICS en particulier via l’outil de gestion d’exécutions multiples MultiSimLib, mais avec l’objectif de rester le plus indépendant possible par rapport aux particularités de STICS, afin que ces outils puissent également être utilisés pour d’autres modèles. Le choix du langage de programmation Matlab et la conception modulaire des outils d’analyse de sensibilité sont les deux seules contraintes qui ont été imposées pour que ces outils soient applicables à STICS dans le contexte de MultiSimLib. Ainsi, ces outils peuvent être appliqués sur tous modèles implémentés en Matlab ou interfacés avec Matlab (cf. la section Principe d’utilisation pour plus de détails sur ce point).

 

Cette bibliothèque est susceptible de contenir des méthodes appartenant à n’importe quelle classe et leur implémentation se fera en fonction des besoins. Les règles d’implémentation sont définies ci-après. La validation de ces outils étant parfois délicate et leur performance étant souvent un point clé dans leur application, un jeu de modèles tests a été implémenté pour offrir un cadre rigoureux de validation et de comparaison de performance des différentes méthodes (cf. la section Validation et tests). Afin d’assurer la maintenance du code et la maîtrise des méthodes implémentées, parfois un peu technique, une documentation détaillée est incluse dans la bibliothèque.

 

 

Architecture de la bibliothèque

 

La bibliothèque est structurée comme suit :

 

- Code : le code de la bibliothèque

 

- Demo : script d’exemple d’utilisation des méthodes implémentées sur les modèles tests

 

- Methods : les fonctions propres aux différentes méthodes

 

- Models : les fonctions implémentant les modèles tests

 

- Tools : les fonctions réalisant des traitements génériques

 

 

 

- Doc : la documentation de la bibliothèque

 

            - Functions : la documentation sur les fonctions générée par M2HTML

 

            - Methods : les documentations propres aux différentes méthodes

 

            - Models : les documentations propres aux différents modèles test

 

            - User : la documentation générale de la bibliothèque

 

 

Principe d’utilisation

 

Le principe d’une analyse de sensibilité est de faire varier une sélection des facteurs d’entrée du modèle étudié et de calculer l’impact de ces variations sur une sélection de ses sorties sous la forme d’un ou plusieurs types d’indices de sensibilité. Bien que cette solution ne soit pas la plus économe en temps de calcul et en mémoire, nous avons choisi de séparer ce processus en deux étapes, c'est-à-dire, pour chaque méthode, il existe deux types de fonctions :

 

-         F_gen_variant_**** : qui génère l’échantillon de valeurs des facteurs d’entrée sur lesquels le modèle doit être appliqué ;

 

-         F_sensi_analysis_**** : qui calcule les indices de sensibilité à partir des valeurs des sorties du modèles.

 

où **** est le nom de la méthode d’analyse de sensibilité.

 

Cette architecture sera commune à l’ensemble des méthodes implémentées dans la bibliothèque. Elle a l’avantage d’être modulaire, et donc d’être facilement applicable quelquesoit le modèle utilisé.

 

Les arguments v_in_mat en sortie de F_gen_variant_**** et v_out_mat en entrée de F_sensi_analysis_****  sont donc commun à l’ensemble de ces fonctions :

 

-         v_in_mat représente la matrice des variants sur lesquels appliquer le modèle, i.e. l’échantillon des valeurs des différents facteurs d’entrée du modèle que l’utilisateur souhaite considérer pour l’analyse de sensibilité, avec en ligne les différents jeux de valeurs générés par la méthode, et en colonne les différents paramètres.

 

-         v_out_mat représente la matrice des sorties du modèle obtenues à partir de la matrice des variant, avec en ligne les différents jeux de valeurs, et en colonne les différentes sorties du modèle que l’utilisateur souhaite analyser.

 

Pour pouvoir utiliser ces fonctions pour un modèle particulier, il est donc nécessaire de pouvoir appliquer le modèle sur les différents jeux de valeurs des paramètres fournis dans la matrice v_in_mat, et de pouvoir copier les valeurs correspondantes des sorties du modèle que l’utilisateur souhaite analyser dans la matrice v_out_mat.

 

Ces fonctions ont bien sûr d’autres arguments qui dépendent de la méthode implémentée. Ces arguments sont largement commentés en entête des fonctions, et nous invitons le lecteur intéressé à se référer à la section de documentation sur les fonctions pour une description détaillée de ces arguments pour chacune des méthodes.

 

Le script progtest.m donne un exemple d’utilisation de ces fonctions pour les modèles tests implémentés. Pour une utilisation sur le modèle STICS, reportez-vous à la documentation de la bibliothèque MultiSimLib.

 

Afin de pouvoir utiliser toutes les fonctions (méthodes, modèles, …) fournies dans la bibliothèque SensiLib, il est nécessaire d’inclure les répertoires contenant ces fonctions au « Matlab search path »,

-         soit via le menu Matlab Fileàset path … ,

-         soit avec l’instruction Matlab addpath(‘répertoire’);

Les répertoires à inclure sont :

-         Code\Methods

-         Code\Demos

-         Code\Tools

-         Code\Models

Validation et tests

 

La validation de l’implémentation des méthodes d’analyse de sensibilité est bien sûre très importante, en particulier car les indices de sensibilité obtenus sur un modèle sont en pratique difficilement vérifiables et pas toujours facilement explicables. Par ailleurs, pour les méthodes dites globales, il convient de vérifier non seulement que les résultats de la méthode convergent vers une valeur exacte, mais également que la vitesse de convergence est correcte, ce qui peut être plus délicat.  

 

La performance des méthodes en terme de nombre de simulation du modèle à réaliser pour obtenir une précision acceptable pour l’utilisateur sur les indices de sensibilité est souvent un aspect important dans le choix de la méthode car celui-ci peut varier de l’ordre de la centaine à l’ordre de la dizaine à centaine de milliers et certaines méthodes peuvent donc se révéler extrêmement coûteuse en temps de calcul selon le temps d’exécution du modèle.

 

Un jeu de modèles tests classiquement utilisés dans la littérature et dont les indices de sensibilité sont connus ou calculables analytiquement a été implémenté pour offrir un cadre rigoureux de validation et de test de performance des méthodes. La liste de ces modèles et leur documentation est donnée ici.

 

Concernant la validation on a essayé, et on essaiera, quand cela est possible, de vérifier les fonctions de génération des variants en traçant, par exemple pour les méthodes globales, les nuages de points résultants et en les comparant avec ceux trouvés dans la littérature et en traçant les distributions de valeurs pour chaque paramètre. L’ensemble de la méthode peut ensuite être vérifiée sur les modèles tests. On s’assure que les valeurs des indices calculés, pour une grande taille d’échantillon pour les méthodes globales, convergent bien vers les valeurs théoriques des indices pour les différents modèles. On essaie également de comparer les vitesses de convergence obtenue avec celles trouvées dans la littérature pour des modèles et conditions d’utilisation identiques.

 

Le jeu de modèle test permet aussi de tester l’influence des éventuels paramètres des méthodes sur les résultats et de comparer les performances des méthodes, c'est-à-dire la précision obtenue sur les indices pour un nombre de simulation équivalent.

 

Tous les résultats de validation et de tests qui ont été obtenus sont rassemblés dans des documents propres à chaque méthode (cf. la section documentation).  

 

Règles d’implémentation

 

Ces règles ont été pour la plupart tirées de celles décrites dans la bibliothèque MultiSimLib.

 

Règles d’implémentation concernant les fonctions :

-         une seule fonction par fichier, le fichier porte le même nom que la fonction ;

-         convention de nommage des fonctions : F_xxxx_yyyy ;

-         les fonctions peuvent être de trois types :

o       fonction propre à une méthode à dans le répertoire Code/Methodes ;

o       fonction ayant un caractère générique à dans le répertoire Code/Outils ;

o       fonction propre à un modèle tests à dans le répertoire Code/Modeles.

La seule exception à cette règle est le programme progtest.m qui permet de tester les méthodes sur les modèles tests (ce programme se trouve dans le répertoire Code).

-         les fonctions qui fournissent les variants et calculent les indices pour une méthode donnée doivent s’appeler respectivement F_gen_variant_*** et F_sensi_analysis_***, où *** est le nom de la méthode ;

-         toutes les fonctions générées pour SensiLib sont construites autour d'un format standard proposant une entête de description des éléments clés de la fonction: arguments en entrée, en sortie, description courte, description longue, méthode d'appel, fonctions appelées. Cf. la section Documentation ci-après pour plus de détails.

 

Règles d’implémentation concernant les variables :

-         convention de nommage des variables : v_xxx_yyy ;

-         éviter autant que possible l’utilisation de variables globales ;

-         les variables contenant les variants et les sorties du modèle doivent s’appeler respectivement v_in_mat et v_out_mat et être des matrices de taille (nombre de simulation, nombre de facteurs variants) et (nombre de simulation, nombre de variables de sorties à analyser)

 

Bien sûr, à tout développement doit être associé une documentation. Cf. la section Documentation ci-après concernant les règles à respecter pour documenter les développements.

 

Documentation

 

Plusieurs types de documentation existent dans SensiLib en plus de celle-ci.

 

La documentation sur les modèles tests :

Pour chaque modèle test, un document décrit au minimum la ou les équations du modèle, sa source et les valeurs des indices de sensibilité associés.  

 

La documentation sur les méthodes d’analyse de sensibilité :

Pour chaque méthode, un document doit donner au moins les éléments suivants :

o       comment se situe la méthode par rapport aux autres (famille de la méthode, avantages, inconvénients, performances) ;

o       comment calculer les indices (en particulier tous les éléments qui ont été nécessaires pour l’implémentation de la méthode).

Le tout en citant toutes les sources d’où sont tirés ces éléments.

 

Pour chaque méthode, un document d’implémentation/validation/tests doit également être rédigé. Ce document doit au minimum contenir les informations suivantes :

o       architecture de l’implémentation de la méthode (noms de toutes les fonctions et description rapide de ce qu’elles réalisent) ;

o       éléments de validation et de test de l’implémentation (cf. section Validation et tests de ce document) ;

 

La documentation dans les fonctions :

Cette partie de documentation est particulièrement importante à la fois pour l’utilisateur et le développeur. C’est elle qui décrit à la fois ce que fait la fonction, comment l’utiliser et comment elle est implémentée.

 

Comme dans MultiSimLib, Chaque fonction doit contenir une entête standard de documentation qui donne les éléments suivants :

o       but de la fonction ;

o       description des arguments d’entrée et de sortie (signification, type, format) ;

o       description du contenu de la fonction ;

o       fonctions appelées ;

o       nom du ou des auteurs ;

o       date de création ;

Un template de cette entête peut être obtenu avec la fonction Code\Functions\Tools\F_sl_new_func.html tirée de MultiSimLib :

o       F_sl_new_func(): création du template générique dans un éditeur ;

o       F_sl_new_func('read_config'): idem avec nom de la fonction ;

o       F_sl_new_func('read_config',1): idem avec création du fichier (répertoire courant) ;

o       F_sl_new_func('read_Config','my_Path'): idem dans un répertoire de son choix.

L’intérêt de ce formattage, outre de donner une certaine homogénéité aux commentaires, est que l'outil M2HTML, utilise ces entêtes des fichiers fonctions pour créer des pages HTML à partir du code matlab, en présentant les informations sous forme didactique. La documentation utilisateur et développeur est ainsi créé automatiquement, et propose une page d'index à travers laquelle on peut parcourir les répertoires de stockages des fonctions, et accéder aux descriptions et au code de ces dernières. C’est la fonction Code\Functions\Tools\F_gen_doc_functions.m qui permet de générer automatiquement cette documentation.

 

Par ailleurs, les différentes étapes de la fonction, les astuces de codage ou autres passages un peu complexes doivent être commentés dans le corps de la fonction. Ces commentaires peuvent s’appuyer sur les autres documents, en particulier pour les fonctions relatives aux méthodes d’analyse de sensibilité, au document de description de la méthode implémentée.
